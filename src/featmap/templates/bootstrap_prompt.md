Scan this project and fill in MAP.md - the project feature map.

1. Identify 2-5 layers. A layer is an architectural zone of responsibility,
   not a folder (examples: "CLI", "Domain core", "Integrations", "Storage").
2. For each layer, find its features. A feature is a user-facing capability
   or a cross-cutting mechanism. Example of the right granularity:
   "Usage limits - trims context and cost to fit model limits".
   Too fine-grained = repeats the code. Too coarse = useless.
3. Fill in MAP.md strictly following the format below. Describe only "what"
   and "where", never "how": implementation details live in the code.
4. If the `featmap check` command is available, run it and fix all errors.

MAP.md format (featmap v1):

```markdown
<!-- featmap v1 -->
# Project: <name>

<2-3 lines: what this project is. The top of the pyramid.>

## Layer: <layer name>

<1-2 lines: what the layer is responsible for.>

### <Feature name> {#kebab-case-id}

**What:** <1-2 sentences: what the feature does.>
**Files:** `path/to/entry.py`, `path/to/other.py:42`
**Depends:** [Another feature](#other-feature-id), or — when there are none
**Status:** active | wip | deprecated
```

Hard format rules:
- The first line of the file is the `<!-- featmap v1 -->` marker.
- Exactly one `#`; `##` only for layers (`## Layer: ...`); `###` only for features.
- Every feature has a unique `{#kebab-case-id}` anchor; no headings deeper than `###`.
- Fields in the order What/Files/Depends/Status; feature body is at most 6 lines.
- Paths in `**Files:**` are relative to the repo root, in backticks.
- Do not add the `**Used by:**` line - it is generated by `featmap links`.
